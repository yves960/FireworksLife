# 如何逐步实现并优化一个 AI Agent Skill

以 flow-trace（跨微服务调用链分析工具）的演进为例

## 背景

flow-trace 是一个帮助分析 Java 微服务调用链的 AI Agent Skill。它从简单的代码分析工具起步，经过 30+ 次迭代，逐步演变成一个功能完善、体验流畅的分析助手。

## 第一阶段：从 0 到 1

### 初始版本

第一个版本很简单：输入一个入口点，分析代码，输出调用链。功能单一，但能用。

```markdown
# flow-trace
输入入口点 → 分析代码 → 输出 JSON
```

### 第一次重构：AI 驱动

很快发现一个问题：硬编码的分析规则太死板，遇到不同的代码风格就歇菜。于是重构为「AI 驱动模式」——让 AI 自己看代码、理解逻辑，而不是靠正则匹配。

**教训**：AI 擅长理解上下文，不要用传统规则限制了它。

## 第二阶段：解决边界问题

### 超时与重试

早期版本经常遇到长调用链超时的问题。解决方案：
- 分批处理，每次只分析一个深度
- 添加超时重试机制
- 增量式结果返回

### 错误恢复

遇到分析失败时，不要直接报错，而是：
1. 记录失败点
2. 返回已分析的部分
3. 给出继续分析的提示

## 第三阶段：体验优化

### 流式输出

用户不喜欢等待。用流式输出（Streaming）让用户看到分析进度：

```
正在分析: UserService.getUser() ✓
正在分析: UserRepository.findById() ✓
...
```

### 交互式修正

AI 分析可能出错，添加「指出错误 → AI 修正」的交互：

```markdown
❓ 在这里停住了？
→ 输入 "continue" 继续
→ 输入 "skip" 跳过
→ 输入具体问题
```

## 总结

做 AI Agent Skill 的心得：

1. **从简单开始**：先跑通核心流程
2. **AI 优先**：让 AI 理解，不要用规则限制它
3. **用户视角**：想想用户实际会怎么用
4. **迭代改进**：30% 的功能用 70% 的时间打磨体验
